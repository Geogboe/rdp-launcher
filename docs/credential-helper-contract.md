# Credential Helper Contract

## Purpose

This document defines the first implementation contract between RDP Launch and an external credential helper. The helper exists so the app can resolve credentials at launch time without becoming a secret store.

This is the contract to build against unless a later ADR replaces it.

## Non-Goals

- The helper is not a profile source.
- The helper is not a configuration registry.
- The helper is not responsible for writing `.rdp` files or launching `mstsc.exe`.
- The helper does not need to support a UI protocol; stdin/stdout JSON is sufficient.

## Transport

- RDP Launch resolves the configured helper executable to a concrete filesystem path before spawn and rejects missing, non-file, or non-runnable paths up front.
- Requests are written as a single JSON object to stdin.
- RDP Launch closes stdin after writing the request so helpers that read until EOF can complete deterministically.
- Responses are read as a single JSON object from stdout.
- Human-readable diagnostics may be written to stderr, but the app must treat stderr as untrusted text and never surface it raw if it could contain secrets.
- The helper must exit after writing its response.

## Timeouts

- `probe`: 3 seconds default timeout.
- `resolve`: 10 seconds default timeout.
- Timeout is treated as a helper failure, not a user cancellation.

## Operations

The first contract supports exactly two operations:

- `probe`
- `resolve`

Anything else is out of scope for the first slice.

## Request Envelope

Every request uses this envelope:

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "op": "resolve",
  "sent_at": "2026-03-07T20:15:00Z",
  "payload": {}
}
```

Fields:

- `version`: integer protocol version. Start at `1`.
- `request_id`: opaque per-request identifier generated by RDP Launch.
- `op`: `probe` or `resolve`.
- `sent_at`: UTC timestamp for correlation.
- `payload`: operation-specific object.

## `probe` Request

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "op": "probe",
  "sent_at": "2026-03-07T20:15:00Z",
  "payload": {}
}
```

### `probe` Response

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "ok": true,
  "payload": {
    "helper_name": "corp-vault",
    "helper_version": "1.2.0",
    "supports": ["probe", "resolve"]
  }
}
```

Use `probe` for the desktop diagnostics view and CLI `helper probe`.

## `resolve` Request

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "op": "resolve",
  "sent_at": "2026-03-07T20:15:00Z",
  "payload": {
    "profile": {
      "id": "profile_01",
      "name": "SQL Cluster - Primary"
    },
    "target": {
      "host": "sql-pri-01.corp.example",
      "port": 3389
    },
    "preset": {
      "id": "preset_admin",
      "name": "Admin"
    },
    "requested_fields": ["username", "password", "domain"],
    "launch_context": {
      "surface": "desktop",
      "reason": "user_launch",
      "allow_windows_vault_bridge": true
    }
  }
}
```

### `resolve` Response

Successful resolution:

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "ok": true,
  "payload": {
    "result": "resolved",
    "credentials": {
      "username": "CONTOSO\\admin-user",
      "password": "secret",
      "domain": "CONTOSO"
    },
    "ttl_seconds": 60,
    "display_message": "Credential resolved from corp-vault"
  }
}
```

Intentional user cancellation:

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "ok": true,
  "payload": {
    "result": "cancelled",
    "display_message": "User cancelled helper prompt"
  }
}
```

Explicit deny:

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "ok": true,
  "payload": {
    "result": "denied",
    "display_message": "Policy denied credential retrieval for this target"
  }
}
```

Helper failure:

```json
{
  "version": 1,
  "request_id": "01HXYZ...",
  "ok": false,
  "error": {
    "code": "backend_unavailable",
    "message": "Vault service unreachable"
  }
}
```

## Result Semantics

Allowed `resolve` result values:

- `resolved`
- `cancelled`
- `denied`
- `prompt`

Meaning:

- `resolved`: helper returned usable credentials.
- `cancelled`: user or helper UI cancelled intentionally.
- `denied`: helper refused on policy or target grounds.
- `prompt`: helper intentionally declines and tells RDP Launch to continue with interactive MSTSC prompting.

If `ok` is `false`, treat the operation as a helper failure rather than a user decision.

## App Behavior Rules

- Nonzero exit code, malformed JSON, missing required fields, or timeout are helper failures.
- Helper failures may fall back to prompt-only launch only if the profile allows prompting.
- `cancelled` never falls through silently into prompt mode; it aborts the launch.
- `denied` never falls through silently into prompt mode; it aborts the launch and surfaces the helper message.
- `prompt` may continue to launch without app-provided credentials.
- If `resolved.password` is present and the profile allows the Windows Credential Manager bridge, the Windows integration layer may create a temporary target-scoped credential for MSTSC compatibility and must clean it up after the launched session exits.

## Security Rules

- RDP Launch must never persist `password` or other returned secret material in SQLite, logs, crash reports, or analytics.
- The helper should return only requested fields.
- The app should avoid echoing raw helper stderr or raw JSON on screen.
- `ttl_seconds` is advisory and intended for cleanup urgency, not long-term caching.

## Why This Contract

- It is simple enough to implement early.
- It supports both desktop and CLI flows.
- It cleanly distinguishes user cancellation from helper failure.
- It keeps password-manager integration out of the core app architecture.

## Deferred

- Listing credential candidates.
- Returning multiple accounts for a target.
- Streaming or interactive sub-protocols.
- Bidirectional helper capabilities beyond `probe`.
